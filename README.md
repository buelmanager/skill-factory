# Skill Factory

Claude Code 스킬을 만들고 테스트하는 작업 공간.

## 핵심 원칙

**스킬 작성은 프로세스 문서에 적용하는 TDD다.**

```
실패하는 테스트 없이는 스킬도 없다 (Iron Law)
```

스킬을 먼저 쓰고 나중에 테스트하면 안 된다. 베이스라인(스킬 없이 에이전트가 실패하는 모습)을 먼저 확인하고, 그 실패를 고치는 최소한의 스킬을 쓴 뒤, 허점을 막는다.

## 디렉터리 구조

```
skill-factory/
├── README.md            # 이 파일 — 제작 워크플로 가이드
├── skills/              # 완성된 스킬 (스킬 1개 = 폴더 1개)
│   └── <skill-name>/
│       ├── SKILL.md     # 필수
│       └── ...          # 무거운 레퍼런스나 재사용 도구만 별도 파일
├── templates/           # SKILL.md 템플릿
└── test-scenarios/      # 스킬별 압박 시나리오와 테스트 기록
    └── <skill-name>.md
```

검증이 끝난 스킬은 `~/.claude/skills/<skill-name>/`으로 복사해 배포한다.

## 제작 워크플로 (RED → GREEN → REFACTOR)

### 1. RED — 베이스라인 실패 확인
1. `test-scenarios/<skill-name>.md`에 압박 시나리오 작성 (규율 스킬은 시간 압박 + 매몰 비용 + 피로 등 3개 이상 결합)
2. 스킬 **없이** 서브에이전트에게 시나리오 실행
3. 에이전트의 선택과 합리화(변명)를 **원문 그대로** 기록

### 2. GREEN — 최소 스킬 작성
1. `templates/SKILL-template.md`를 `skills/<skill-name>/SKILL.md`로 복사
2. RED에서 기록한 구체적 실패만 다루는 최소 내용 작성
3. 같은 시나리오를 스킬과 **함께** 실행해 통과 확인

### 3. REFACTOR — 허점 봉쇄
1. 테스트 중 나온 새 합리화에 명시적 반박 추가 (합리화 표 + Red Flags 목록)
2. 통과할 때까지 재테스트

## 작성 규칙 요약

- `name`: 영문 소문자·숫자·하이픈만, 동사형(-ing) 선호 (예: `creating-skills`)
- `description`: "Use when..."으로 시작, **발동 조건만** 기술 — 워크플로 요약 금지 (요약하면 에이전트가 본문을 읽지 않고 description만 따라간다)
- frontmatter 전체 1024자 이내, 3인칭 서술
- 검색 키워드(에러 메시지, 증상, 도구명)를 본문 곳곳에 배치
- 예제는 한 언어로 훌륭한 것 하나만
- 일반 스킬 500단어 이내, 자주 로드되는 스킬 200단어 이내
- 다른 스킬 참조는 이름만 (`@` 링크는 강제 로드라 금지)

상세 기준은 `superpowers:writing-skills` 스킬 참조.
